# Bugs rencontrés — 24 juin 2026 — Linux AppImage

## BUG 1 — `--no-bundle` inexistant dans Tauri v1
`tauri build --no-bundle` → erreur immédiate. Flag absent de Tauri v1.
**Fix :** `cargo build --release --features custom-protocol`

## BUG 2 — `@yao-pkg/pkg` produit des binaires corrompus
Sidecar backend crashait : `SyntaxError: Invalid or unexpected token` dans `bootstrap.js`.
Corrompu avec node18/20/22.
**Fix :** Node.js SEA — injecte le bundle dans le binaire `node` natif de la CI.

## BUG 3 — `binaries/` absent du checkout (gitignored)
`cp "$NODE_BIN" "$OUT"` échouait : "No such file or directory".
**Fix :** `mkdir -p "$BINDIR"` avant toute écriture dans le step sidecar.

## BUG 4 — Sidecar sans triple → app crash au démarrage
Tauri v1 cherche `collab-backend-x86_64-unknown-linux-gnu` au runtime.
Le workflow copiait `collab-backend` sans triple → introuvable, backend ne démarre jamais.
**Fix :** Conserver le triple dans tous les artefacts (AppImage, ZIP, tarball macOS).

## BUG 5 — Heredoc à colonne 0 casse le parser YAML
```yaml
cat > file << 'EOF'
{   # parser YAML interprète { comme un mapping YAML, pas du shell
```
Workflow échouait en 0s.
**Fix :** `printf '{"main":"..."}' > sea-config.json`

## BUG 6 — linuxdeploy exit code 8
`wget -q` retourne exit code 8 (erreur HTTP) sur l'URL du plugin GTK. Bash `-e` tue le script.
**Fix :** `curl --fail` + suppression du plugin GTK.

## BUG 7 — WebKitNetworkProcess absent de l'AppImage [EN COURS]
`libwebkit2gtk-4.0.so` cherche ses helpers à un path absolu **hardcodé dans le binaire** :
`/usr/lib/x86_64-linux-gnu/webkit2gtk-4.0/`
Ubuntu 24.04 n'a que webkit2gtk-4.1. `WEBKIT_EXEC_PATH` non supporté par cette version.
**Fix (v1.2.0) :** `tauri build --bundles appimage` — bundler officiel Tauri qui gère webkit
via son fork du plugin GTK (`tauri-apps/linuxdeploy-plugin-gtk`).
